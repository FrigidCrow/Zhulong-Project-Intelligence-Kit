Command Manual

AI-PIKit 命令控制台。

这里明确每个命令的用途、典型用法、产出文件，以及接手新项目和既存项目时的 step-by-step 操作。

先用命令地图建立方向感。

通用规则

安装 bin 后推荐直接使用 `pik-*`。未安装 bin 时，可以在 AI-PIKit 仓库里使用 `node bin/pik.mjs `。

pik-docs-scan --target /path/to/repo
node bin/pik.mjs docs scan --target /path/to/repo
参数用途注意
--target <repo>指定目标项目目录。大多数命令都需要。
--template选择初始化模板。常用 `greenfield-app`、`brownfield-monorepo`。
--mode new|existing写入项目接入模式。既存项目建议显式写 `--mode existing`。
--run执行配置中的外部命令。可能触发 GraphRAG/Graphify 的联网行为，取决于配置。
--rag执行配置的 RAG query command。结果写入 `RAG_QUERY_RESULT.md`。
--writeback把 evidence 摘要追加到目标记录。目标路径必须在项目内。

新项目 step-by-step

新项目可以从第一天开始积累 `.planning/`，先建立 workspace，再逐步补文档、代码地图和 evidence。

cd /path/to/new-project

pik-init --target "$PWD" --template greenfield-app --name my_new_project --mode new
pik-codebase-scan --target "$PWD"
pik-docs-scan --target "$PWD"
pik-verify --target "$PWD"

pik-new-milestone --target "$PWD" "MVP1 walking skeleton"
pik-plan-phase --target "$PWD" "phase 1"
pik-execute-phase --target "$PWD" "implement first vertical slice"
pik-verify-work --target "$PWD" "verify phase 1"

既存项目 step-by-step

既存项目不要大改原结构。AI-PIKit 作为本地 intelligence layer 叠加进去。初始化后必须先跑 codebase scan。

cd /path/to/existing-project

pik-init --target "$PWD" --template brownfield-monorepo --name existing_project --mode existing
pik-codebase-scan --target "$PWD"
pik-codebase-status --target "$PWD"

pik-docs-scan --target "$PWD"
pik-docs-normalize --target "$PWD"
pik-docs-query --target "$PWD" "障害 承認 QA"

pik-graph-build --target "$PWD" --run
pik-graph-status --target "$PWD"

pik-debug --target "$PWD" "生产审批金额异常"
既存项目的 `codebase` gate 要求 `.planning/codebase/CODEBASE_STATUS.md` 存在，并且源码数量大于 0。

Refresh / Mode 命令

这些命令用于控制执行成本和上下文新鲜度。它们默认只做轻量检查和提醒；真正刷新 GraphRAG / Graphify 必须显式调用 `pik-refresh-run`，或使用已有显式命令 `pik-docs-index --run`、`pik-graph-build --run`。

命令用途示例产出
pik-preflight检查 RAG/Graphify 距离当前 commit 落后多少，以及是否有相关变更。pik-preflight --target "$PWD"PREFLIGHT.md, PREFLIGHT.json
pik-refresh-plan生成差分刷新建议；无关 commit 会建议跳过。pik-refresh-plan --target "$PWD"REFRESH_PLAN.md, REFRESH_PLAN.json
pik-refresh-run显式执行 RAG、Graphify 或两者的刷新。pik-refresh-run --target "$PWD" --graphREFRESH_RUN.md, REFRESH_STATE.json
pik-mode-status查看当前执行预算模式。pik-mode-status --target "$PWD"MODE.md
pik-mode-set切换 `default-local-rag`、`graph-lite`、`full-strict`。pik-mode-set --target "$PWD" graph-lite.planning/config.json, MODE.md
`pik-preflight` / `pik-refresh-plan` 一定不会执行重刷新。看到 `behind-unrelated` 时，说明落后的 commit 没改到文档源或代码地图相关文件，可以先跳过。

MVP3 证据质量 / Trace / Policy 命令

MVP3 的重点是把 RAG 结果、citation、代码影响面、测试和 evidence 串成可审计链路。日常完成前建议至少跑一次 `pik-policy-check --strict`。

命令用途示例产出
pik-trace-build把文档索引、Graphify graph、测试文件和 evidence 记录汇总成 trace matrix。pik-trace-build --target "$PWD"TRACE_MATRIX.json, TRACE_MATRIX.md
pik-trace-query按关键词查询 trace matrix，快速确认某条仕様关联了哪些代码、测试和证据。pik-trace-query --target "$PWD" "代理承認"stdout 查询结果。
pik-trace-audit检查 trace matrix 是否具备文档、代码和 evidence 维度。pik-trace-audit --target "$PWD"TRACE_AUDIT.md
pik-policy-list列出当前内置 policy：local-only、citation、golden、trace、graph freshness、license。pik-policy-list --target "$PWD"POLICY_LIST.md
pik-policy-explain解释单条 policy 的目的和建议命令。pik-policy-explain --target "$PWD" trace.matrixstdout 解释。
pik-policy-check执行所有 policy，并把四态结果写成报告。pik-policy-check --target "$PWD" --strictPOLICY_CHECK.md, POLICY_CHECK.json
pik-policy-lock在 offline lock 后生成稳定 policy snapshot 和 SHA-256 hash。pik-policy-lock --target "$PWD"POLICY_LOCK.md, POLICY_LOCK.json
pik-policy-verify对比当前配置和 lock，同时执行轻量 privacy/preflight/citation/graph freshness checks。pik-policy-verify --target "$PWD"POLICY_VERIFY.md, POLICY_VERIFY.json
pik-policy-diff输出 lock snapshot 与当前 snapshot 的字段级差异。pik-policy-diff --target "$PWD"POLICY_DIFF.md, POLICY_DIFF.json
pik-help-skills根据“我现在是什么情况”推荐适合的 `pik-*` 命令组。pik-help-skills --target "$PWD" "文档更新后想确认影响面"HELP_SKILLS.md

Workflow guard 命令

Workflow guard 会检查 context、codebase、docs、graph、plan、implementation、verification、evidence、writeback。证据不齐时，completion check 返回非 0。

命令用途示例
pik-workflow-run启动任意 guard workflow。pik-workflow-run --target "$PWD" debug "bug"
pik-workflow-status查看 active workflow gate 状态。pik-workflow-status --target "$PWD"
pik-workflow-continue标记人工 gate。pik-workflow-continue --target "$PWD" --gate plan --evidence "PLAN.md reviewed"
pik-workflow-audit输出失败 gate 原因和下一条建议命令。pik-workflow-audit --target "$PWD"
pik-gate-check检查所有 gate。pik-gate-check --target "$PWD"
pik-completion-check最终完成前检查。pik-completion-check --target "$PWD"

Public workflow 命令

日常开发优先用这一组命令。它们会自动写 workflow facade，汇总 preflight、policy、docs、graph、evidence 和下一步建议，但不会自动触发 GraphRAG index、Graphify build 或 refresh-run。

命令场景内部路由
pik-new-milestone开启新 milestone。workflow run new-milestone
pik-spec-phase梳理需求和仕様依据。workflow run spec-phase
pik-discuss-phase讨论决策和未决点。workflow run discuss-phase
pik-ui-phase梳理 UI 范围和验证。workflow run ui-phase
pik-debug缺陷调查、根因分析和验证。workflow run debug
pik-plan-phase制定阶段计划。workflow run plan-phase
pik-execute-phase执行改修或新规开发。workflow run execute-phase
pik-code-review代码审查和影响面确认。workflow run code-review
pik-verify-work验证实现是否满足需求。workflow run verify-work
pik-complete-milestone完成 milestone 并整理证据。workflow run complete-milestone

全命令详解

这一节按命令为单位展开。判断一个命令是否真的生效，不看聊天里说了什么，而看它是否产生了对应本地 artifact，或是否让 guard 状态发生变化。

pik

总入口
用途
AI-PIKit CLI 的统一入口。安装 bin 后通常直接用具体命令；调试时可用 `node bin/pik.mjs`。
何时用
查看 help、验证命令路由、在未安装全局 bin 时从仓库内运行。
判断标准
`pik --help` 能列出全部命令；`node bin/pik.mjs --help` 输出一致。
pik --help
node bin/pik.mjs --help

pik-init

初始化
用途
把 AI-PIKit intelligence layer 叠加到目标项目，不移动原源码。
何时用
新项目第一步、既存项目接入第一步。既存项目后面必须立刻跑 `pik-codebase-scan`。
产出
`AGENTS.md`、`project.manifest.yml`、`.planning/`、`.planning/INIT_PROFILE.md`。
pik-init --target "$PWD" --template brownfield-monorepo --name existing_project --mode existing
cat .planning/INIT_PROFILE.md

pik-verify

初始化检查
用途
检查目标项目是否具备 AI-PIKit 必要文件和目录。
何时用
init 后、复制项目后、怀疑 `.planning/` 缺失时。
判断标准
stdout 中关键项为 `ok`；缺失项会显示 `missing`。
pik-verify --target "$PWD"

pik-map

轻量地图
用途
生成一个轻量项目结构说明，适合快速看目录和入口。
何时用
临时理解项目结构，或在完整 codebase scan 前快速建初始结构。
产出
`.planning/codebase/STRUCTURE.md`。
pik-map --target "$PWD"
sed -n '1,80p' .planning/codebase/STRUCTURE.md

pik-codebase / pik-codebase-scan

代码基线
用途
扫描源码、测试、配置、入口线索，建立 codebase baseline。
何时用
既存项目 init 后必须先跑；每次大范围代码变更、目录调整、框架迁移后重跑。
产出
`CODEBASE_STATUS.md`、`STRUCTURE.md`、`STACK.md`、`TESTING.md`、`ARCHITECTURE.md`。
pik-codebase-scan --target "$PWD"
pik-codebase-status --target "$PWD"

pik-codebase-status

代码基线检查
用途
查看 codebase scan 是否存在，以及 Graphify graph 的同步状态。
何时用
开始 `pik-debug`、`pik-plan-phase` 前确认既存项目不是裸接入。
判断标准
应看到 `.planning/codebase/CODEBASE_STATUS.md` 等文件为 ok。
pik-codebase-status --target "$PWD"

pik-docs-scan

文档发现
用途
扫描 `docs`、`documents`、`仕様書` 以及配置的文档目录，生成文档来源清单。
何时用
第一次导入文档后、文档新增/删除/移动后、开始需求分析前。
产出
`.planning/knowledge/RAG_SOURCES.md`、`DOC_RAG_STATUS.md`。
pik-docs-scan --target "$PWD"
pik-docs-status --target "$PWD"

pik-docs-normalize

文档归一化
用途
把可读文本类文档复制/转写成稳定的 normalized markdown 入口。
何时用
第一次扫描文档后、文档内容更新后、本地关键词查询前。
产出
`.planning/knowledge/normalized/` 下的规范化文件。
pik-docs-normalize --target "$PWD"
ls .planning/knowledge/normalized

pik-docs-query

本地文档查询
用途
对 normalized 文档做本地关键词查询，不依赖外部 RAG provider。
何时用
想快速查仕様、QA、議事録里的关键词；保密项目优先用这个模式。
判断标准
输出应包含命中文档路径、行号或片段；没有命中则说明本地文档中未发现。
pik-docs-query --target "$PWD" "代理承認 上限"

pik-rag-init-local

默认本地 GraphRAG
用途
创建 `graphrag-workspace/`、同步文档到 `input/`、生成 Ollama + LanceDB 的本地 `settings.yaml`。
何时用
第一次接入项目、文档来源变化后、需要重建本地 GraphRAG 前。
前置
本机可执行 `graphrag`、`ollama`，并已拉取 `qwen2.5:14b` 和 `nomic-embed-text`。
保密
默认不需要 `GRAPHRAG_API_KEY`，不生成 `.env`，只允许 `http://127.0.0.1:11434` / `localhost`。
产出
`graphrag-workspace/settings.yaml`、`graphrag-workspace/input/`、`LOCAL_RAG_STATUS.md`。
pik-rag-init-local --target "$PWD"
sed -n '1,120p' .planning/knowledge/LOCAL_RAG_STATUS.md

pik-privacy-audit

本地保密审计
用途
检查 `.planning/config.json`、`project.manifest.yml`、`graphrag-workspace/settings.yaml` 和 RAG 结果中是否出现外部 endpoint、外部 provider 或 key。
何时用
每次改 RAG 配置后、执行 `pik-docs-index --run` 或 `pik-docs-query --rag` 前。
阻断
local-only 模式下发现外部 URL、`openai/deepseek/azure` provider、`GRAPHRAG_API_KEY` 等会失败。
pik-privacy-audit --target "$PWD"
sed -n '1,120p' .planning/knowledge/PRIVACY_AUDIT.md

pik-docs-index / pik-docs-index --run

RAG 索引
用途
不带 `--run` 时生成 handoff；带 `--run` 时执行 `.planning/config.json` 的 `index_command`。
何时用
运行 `pik-rag-init-local` 后，第一次建索引或文档大更新后。
保密
local-only 模式会先跑隐私审计；外部 provider 配置会被阻断。
产出
`RAG_INDEX_HANDOFF.md`，执行后还有 `RAG_INDEX_RESULT.md`。
pik-docs-index --target "$PWD"
