通用规则

安装 bin 后推荐直接使用 pik-*。未安装全局 bin 时，可以在 AI-PIKit 仓库里使用 node bin/pik.mjs <subcommand>。默认流程必须是 local-only、no hidden heavy refresh；只有显式 --run、--index 或 pik-refresh-run 才允许重任务。

pik-docs-sync --target /path/to/repo
node bin/pik.mjs docs sync --target /path/to/repo

场景路线

先判断项目文档严格程度，再决定是否启用 RAG。默认路线是轻量接入：文档只做参考、RAG 不安装、GraphRAG index 不运行；只有规格强约束项目才进入本地 RAG 路线。

新项目从 0 接入

先接入 AI-PIKit，再安装 runtime pack，然后建立代码和文档轻量基线，最后进入第一次 milestone。默认不安装 RAG；如果项目从一开始就是规格严格型，再切到下面的本地 RAG 路线。

cd /path/to/new-project

pik-init --target "$PWD" --template greenfield-app --name my_new_project --mode new --doc-policy reference --rag none
pik-runtime-install --runtime codex --dest ~/.codex/skills --force
pik-runtime-install --runtime claude-code --dest ~/.claude/skills --force
pik-runtime-install --runtime github-copilot --dest .github/prompts --force

pik-codebase-scan --target "$PWD"
pik-docs-sync --target "$PWD"
pik-graph-build --target "$PWD" --run

pik-new-milestone --target "$PWD" "MVP1 walking skeleton"
pik-spec-phase --target "$PWD" "整理 MVP1 仕様"
pik-plan-phase --target "$PWD" "MVP1 phase 1"
pik-execute-phase --target "$PWD" "实现第一条纵向链路"
pik-verify-work --target "$PWD" "验证 MVP1 phase 1"
pik-complete-milestone --target "$PWD" "MVP1 walking skeleton"

严格文档项目在初始化时改用本地 RAG 路线：

pik-init --target "$PWD" --template greenfield-app --name my_new_project --mode new --doc-policy strict --rag local --setup-rag skip
pik-rag-init-local --target "$PWD"
pik-docs-index --target "$PWD" --run

既有项目接入

既有项目不要移动原源码。AI-PIKit 只叠加 .planning/，先扫描已有代码和文档，再从当前真实改修任务进入第一次 workflow。默认仍然是 reference + rag none，避免一接入就安装 RAG 或重建索引。

cd /path/to/existing-project

pik-init --target "$PWD" --template brownfield-monorepo --name existing_project --mode existing --doc-policy reference --rag none
pik-codebase-scan --target "$PWD"
pik-codebase-status --target "$PWD"

pik-docs-sync --target "$PWD"
pik-graph-build --target "$PWD" --run
pik-preflight --target "$PWD"

pik-debug --target "$PWD" "生产审批金额异常"

如果这个既有项目是对日/规格严格项目，接入时直接选择 --doc-policy strict --rag local --setup-rag skip，再单独执行 pik-rag-init-local 和 pik-docs-index --run。

日常开发循环

日常优先使用 workflow 主循环命令。GraphRAG、Graphify、policy、evidence 会作为 guard 和 next command 被串起来，不需要开发者每次手动想一遍。

pik-new-milestone --target "$PWD" "CR-017 代理承認上限修正"
pik-spec-phase --target "$PWD" "確認仕様と QA"
pik-discuss-phase --target "$PWD" "确认实现策略"
pik-plan-phase --target "$PWD" "拆分实现和验证"
pik-execute-phase --target "$PWD" "实施改修"
pik-verify-work --target "$PWD" "跑测试并记录证据"
pik-complete-milestone --target "$PWD" "CR-017 收口"

文档更新循环

文档更新后默认只跑轻量同步，不自动重建 GraphRAG index。需要重索引时必须显式加 --index，并先确认 local-only 配置没有被改坏。

pik-docs-sync --target "$PWD"
pik-docs-query --target "$PWD" "代理承認 上限"
pik-answer-audit --target "$PWD"

# 只有明确需要重建本地 GraphRAG index 时才运行
pik-privacy-audit --target "$PWD" --strict
pik-docs-sync --target "$PWD" --index

状态处理

AI-PIKit 的报告会尽量给出四态结果。日常不要只看 exit code，还要看状态语义和 next commands。

状态 / 提示什么时候出现怎么处理
PASS对应 gate 或 audit 可以作为完成证据。继续 workflow，并用 pik-evidence-record 写回证据。
WAIVED_WITH_RISKreference 文档策略下缺文档、缺 citation 或 cockpit 缺 artifact。允许继续，但必须在 evidence 或 workflow 报告里写明风险；规格强约束任务应切到 docs-strict。
STALE_NEEDS_REFRESHRAG 或 Graphify 距离相关 commit 已落后。先跑 pik-refresh-plan；确认相关后再显式刷新。
FAIL隐私、外部 provider、missing citation、full-strict stale 等阻断项。打开对应报告，按 next commands 修复后重跑检查。
RAG backend disabled项目初始化为 --rag none，但执行了 GraphRAG index/query。轻量项目改用普通 pik-docs-query；严格项目重新选择本地 RAG 并运行 pik-rag-init-local。

质量验证循环

开发中用轻量 gate，收口时用质量闭环 gate。所有报告都留在本地仓库的 verification/reports/ 或目标项目的 .planning/ 下。

npm run verify:quality
npm run verify:full-command-surface
npm run verify:skills-usability
npm run verify:workflow-closure
npm run verify:cockpit-build
npm run verify:docs-completeness
npm run verify:quality-closure

参数说明
  • --target <repo>: 指定目标项目目录。
  • --template <name>: 选择初始化模板。
  • --name <name>: 写入项目名。
  • --mode new|existing: 指定新项目或既存项目。
  • --doc-policy reference|strict: 选择文档是参考资料还是强约束依据。
  • --rag none|local|external: 选择不启用 RAG、本地 RAG 或外部 RAG。
  • --setup-rag ask|install|skip: 本地 RAG 依赖处理方式；默认 skip，只写 setup plan。
  • --allow-external-rag: 显式确认外部 RAG 可能导致文档内容离开本机。
  • --interactive: 强制进入 init wizard；真实终端只运行 `pik-init --target <repo>` 也会进入向导。
  • --no-interactive: 在真实终端中也跳过向导，使用显式参数或默认值。
  • --force: 覆盖已有模板文件。
默认行为
真实终端默认进入 init wizard；CI/非 TTY 默认使用 `reference + rag none + local_only`。只叠加 `.planning/` 和配置，不安装 RAG、不执行 GraphRAG index、不执行 Graphify build；strict 必须显式选择 local/external RAG。
