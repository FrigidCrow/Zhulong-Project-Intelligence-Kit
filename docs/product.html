普通 AI coding 只看到 prompt 和当前文件。AI-PIKit 让 AI 先读项目状态、仕様依据、QA、議事録、 代码影响面和验证要求，再进入实现。

四个层次，组成一个工程控制面。

AI-PIKit 的价值不是把几个工具绑在一起，而是把它们变成 AI 开发时必须经过的上下文路径。

Workflow guard

用 AI-PIKit native workflow 复刻 GSD 的严格流程思路。`pik-debug`、`pik-plan-phase`、`pik-execute-phase` 都会进入 gate 检查。

Document intelligence

面向对日文档密集型项目，把 仕様書、QA、議事録、设计文档变成可扫描、可查询、可引用的本地知识层。

Code map

Graphify 提供代码地图和影响面。AI-PIKit 负责把图谱产物同步进 `.planning/graphs/`，供 workflow 使用。

Evidence loop

测试命令、验证结果、风险判断、仕様引用都写回 evidence。最后由 `pik-completion-check` 判断能不能收口。

面向对日开发现场的真实摩擦。

AI-PIKit 优先解决“文档很多、旧代码复杂、交付要讲依据”的场景，而不是做一个空泛的聊天外壳。

仕様变更

先把变更点进入 `pik-spec-phase`，查询文档依据，记录未决点，再生成计划。

缺陷调查

`pik-debug` 会写 context packet，把 QA、議事録、Graphify 影响面和验证 gate 串起来。

既存改修

既有项目初始化后必须跑 `pik-codebase-scan`。没有 codebase baseline，guard 不允许轻易完成。

文档更新

文档新增或变更后先跑 `pik-docs-sync`。它只标记 `STALE_NEEDS_REFRESH`，不自动重建 GraphRAG；需要新索引时再显式加 `--index`。

代码更新

别人提交和你当前任务无关时可以跳过重刷新；相关源码或测试变更才推荐 `pik-refresh-run --graph`。

AI-PIKit 的工作方式

它不搬动你的项目，也不替代 IDE。它在目标项目根目录叠加一个本地工程控制面。

1. pik-debug / pik-plan-phase
启动 AI-PIKit workflow guard。
2. .planning/context
生成 context packet 和 handoff。
3. docs + graph
查询文档依据和代码影响面。
4. evidence
把验证记录写入 `.planning/evidence/` 并回写 issue。
5. completion-check
所有 gate PASS 才允许完成。

AI-PIKit 的产物默认留在目标项目本地。外部流通只会来自你配置的 runtime、RAG provider 或 Graphify command。

每一步都有落点，不只停在聊天里。

AI-PIKit 会把 AI 开发过程拆成可检查的本地文件。你可以打开这些文件，确认它到底有没有接入文档、Graphify 和 workflow。

.planning/codebase/代码清单、结构、技术栈、测试策略、既有项目 baseline。
.planning/knowledge/文档来源、归一化结果、RAG handoff、RAG query result。
.planning/graphs/Graphify graph、报告、diff、staleness 检查。
.planning/workflows/workflow state、gate PASS/FAIL、人工继续记录。
.planning/evidence/验证证据、命令输出摘要、source 引用、evidence index。
.planning/quality/RAG golden、citation audit、RAG eval，用来证明文档问答结果可信。
.planning/trace/文档、代码、测试、evidence 的追踪矩阵。
.planning/policies/完成前 policy check，保密和证据质量统一给出 PASS/FAIL。
.planning/refresh/记录 GraphRAG / Graphify 上次刷新 commit、preflight、refresh plan 和显式刷新结果。
AGENTS.md把 AI-PIKit 调用规则写进目标项目，让 runtime 更容易遵守流程。

成本可控，不把每个任务都变成全量重建。

MVP3.5 增加执行预算层。AI-PIKit 会告诉你 GraphRAG / Graphify 距离最新 commit 落后多少，以及这些 commit 是否真的改到了相关文档或代码。

轻量提醒

pik-preflight --target "$PWD"
sed -n '1,120p' .planning/refresh/PREFLIGHT.md

只读 git diff 和刷新账本，输出 `heavy refresh executed: no`。

差分计划

pik-refresh-plan --target "$PWD"
sed -n '1,160p' .planning/refresh/REFRESH_PLAN.md

无关 commit 显示 `behind-unrelated`；相关文档/源码才推荐刷新。

显式刷新

pik-refresh-run --target "$PWD" --rag
pik-refresh-run --target "$PWD" --graph
pik-mode-set --target "$PWD" docs-strict

真正刷新必须由命令触发，结果写回 `REFRESH_STATE.json`。

适合的项目

AI-PIKit 的目标用户不是纯玩具项目，而是代码、文档、变更依据和交付责任同时存在的开发现场。

对日开发

仕様書、QA、議事録、画面设计、测试规格书都能成为 AI 开发依据。

既有系统改修

先建立 codebase baseline，再用 Graphify 看影响面，避免凭感觉改旧系统。

长周期交付

用 evidence 和 writeback 记录每次判断、验证、风险和后续事项。

可信度来自可复跑验证。

AI-PIKit 不要求你相信文案。它要求每个接入点都有命令、有产物、有验证报告。

接入验证

`pik-verify --target "$PWD"` 检查目标项目是否真的有 `AGENTS.md`、`project.manifest.yml`、`.planning/`。

能力验证

`npm run verify:integration` 覆盖 init、docs、Graphify、runtime pack、workflow guard 和 evidence；`npm run verify:init-policy` 覆盖 `reference/strict` 文档策略、`none/local/external` RAG 后端和外部 RAG opt-in；`npm run verify:knowledge-reliability` 覆盖 docs sync、docs query、answer audit；`npm run verify:mvp3` 覆盖 golden、citation、trace、policy、help skills；`npm run verify:mvp35` 覆盖 refresh/preflight/mode 控制；`npm run verify:workflow-facade` 和 `npm run verify:policy-hardening` 覆盖无感编排层和策略合同。

命令面验证

`npm run verify:full-command-surface` 会逐个执行 `package.json` 中全部 `pik-*` / `pik` bin 命令，避免“文档写了但实际不能用”。

项目驾驶舱

`pik-cockpit-build` 会把 Graphify 影响图、GraphRAG/RAG 证据链、workflow、quality 和 privacy 状态生成成本地静态 HTML，适合 leader 演示。稳定样例页在 `templates/cockpit/sample.html`，真实项目快照在 `.planning/cockpit/index.html`。

开发者审计 / 对标

`npm run dev:audit:full` 会生成维护者专用 scorecard：71 个命令、33 个 runtime skill/prompt、feature gate、AI-PIKit / GSD / Superpowers 对标、耗时和 token 边界。最近一次完整审计为 96 / A，三方 benchmark 为 AI-PIKit 90 / A、GSD 88 / B、Superpowers 82 / B。Scorecard 里的 Benchmark comparison 87 是混合对标平均分，不是 AI-PIKit 单体分。摘要在 `verification/reports/developer-audit-summary.md`。

完成验证

`pik-completion-check` 会在 evidence 或 writeback 缺失时阻断，避免“看起来完成了”的假收口。

从第一条命令开始

新项目和既存项目都可以接入。区别是既存项目必须先扫描现有代码库，给 guard 建立 codebase baseline。

新项目

pik-init --target "$PWD" --template greenfield-app --name my_new_project --mode new --doc-policy reference --rag none
pik-codebase-scan --target "$PWD"
pik-preflight --target "$PWD"
pik-verify --target "$PWD"
