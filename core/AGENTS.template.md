## Zhulong Command Facade

When the user invokes Zhulong commands such as `$zl-debug`, `$zl-execute-phase`,
`$zl-plan-phase`, or `$zl-code-review`, treat the command as a Zhulong-enhanced
engineering flow.

The user should not need to manually run separate document, graph, and evidence
commands before using the workflow. Instead, perform the relevant context
checks inside the Zhulong command flow.

Zhulong uses a native workflow kernel. GSD is reference design only, not the
required backend and not the public command namespace.

Core public workflow commands:

| Public Zhulong command | Workflow purpose |
| --- | --- |
| `$zl-new-milestone` | Start a milestone loop |
| `$zl-spec-phase` | Clarify requirements and source evidence |
| `$zl-discuss-phase` | Resolve decisions, options, and open questions |
| `$zl-ui-phase` | Define UI scope, states, contracts, and verification |
| `$zl-debug` | Diagnose and fix defects |
| `$zl-plan-phase` | Plan implementation with evidence and impact surface |
| `$zl-execute-phase` | Implement planned work |
| `$zl-code-review` | Review changes against evidence and impact |
| `$zl-verify-work` | Verify completed work |
| `$zl-complete-milestone` | Close a milestone with evidence and follow-ups |

### Required Flow

1. Read Zhulong state first:
   - `.planning/STATE.md`
   - active phase records under `.planning/phases/`
   - active issue or debug records under `.planning/issues/` or `.planning/debug/`
2. Check specification context for business, requirement, screen, API, DB,
   QA, test, terminology, or customer-intent claims:
   - `.planning/knowledge/RAG_SOURCES.md`
   - `.planning/knowledge/DOC_RAG_STATUS.md`
   - `.planning/knowledge/GLOSSARY.md`
   - `.planning/knowledge/REQUIREMENT_TRACE.md`
   - CLI helpers: `zl-docs-scan`, `zl-docs-normalize`,
     `zl-docs-index`, `zl-docs-index --run`, `zl-docs-query`, and
     `zl-docs-query --rag`
   - GraphRAG or another document RAG adapter if configured and available
3. Check code-map context before risky edits:
   - `.planning/graphs/GRAPH_REPORT.md`
   - `.planning/graphs/graph.json`
   - Graphify query/path/explain output if configured and available
   - CLI helpers: `zl-graph-status`, `zl-graph-query`,
     `zl-graph-build`, `zl-graph-build --run`, and `zl-graph-diff`
   - source search with `rg` and direct file reads
4. Create or update a context packet under `.planning/context/` when the work
   is substantial, unclear, or high-risk. CLI helpers: `zl-context-debug` and
   `zl-context-execute`.
   - Public workflow helpers such as `zl-debug` and `zl-execute-phase` create
     both a context packet and a Zhulong native workflow handoff record.
5. Create or refine the plan before implementation.
6. Implement with the coding agent.
7. Verify with focused tests, build checks, source assertions, logs, or manual
   checks appropriate to the task.
8. Write back evidence, decisions, risks, and follow-ups to the relevant
   `.planning/` record. CLI helpers: `zl-evidence-record` and
   `zl-evidence-status`.
   - Use `zl-evidence-record --writeback <record>` to append evidence to an
     existing issue, debug, or phase record.

## Situation Routing

| Situation | Required action |
| --- | --- |
| Bug report with an error, class, function, route, screen, or job | Use `$zl-debug` flow. Read state, check relevant spec/QA/minutes, then use code-map/source search for impact and root cause. |
| New feature or change request | Use `$zl-plan-phase` or `$zl-execute-phase` flow. Confirm specification evidence before planning implementation. |
| Business rule, acceptance condition, screen behavior, API contract, DB rule, or terminology question | Check `.planning/knowledge/`, `zl-docs-query`, and configured document RAG before answering. Cite source documents or mark as hypothesis. |
| Call-chain, dependency, entry point, or impact question | Use `zl-graph-query` or the configured code-map adapter, then verify against source files. |
| Before editing a high-impact module | Do an impact check with `zl-graph-status`, code-map artifacts, and source search. |
| After editing code | Run focused verification and refresh graph data if structural relationships changed. Use `zl-graph-build --run` when direct Graphify execution is approved, or `zl-graph-build` to create a backend handoff. |

## Workflow Handoff Pattern

When a `zl-*` workflow helper creates `.planning/context/handoffs/*-HANDOFF.md`,
read that file and its linked context packet before continuing the workflow.
Preserve the public `zl-*` command in user-facing notes. If a handoff mentions
`gsd-*`, treat it as historical reference design only.

When suggesting follow-up commands, reminders, or next actions to the user, use
the public `zl-*` command names. Do not tell the user to run `gsd-*` directly;
`gsd-*` is not the active Zhulong command surface.

## Evidence Rules

- Treat graph and RAG output as orientation, not final proof.
- Verify claims against source documents, source files, SQL/schema, tests, logs,
  API responses, or other primary evidence.
- Do not present document RAG output as fact unless the source document or
  citation has been checked.
- Record important findings in the relevant issue, debug, phase, or knowledge
  file.
- For non-trivial work, create or update a durable evidence record with
  `zl-evidence-record`.
- Link evidence back to the active issue, debug, or phase record with
  `--writeback`, `--issue`, `--debug`, or `--phase` when a backend record exists.
- Do not copy secrets, tokens, private credentials, or production data into
  planning records.

- Bugs: diagnose first, then patch only after root cause is clear.
- Features: confirm specification context, then plan or execute.
- Reviews: prioritize correctness, specification drift, security, test gaps,
  dependency direction, and architecture boundaries.
- Verification: include commands run, results, remaining risk, source evidence,
  and rollback notes when relevant.
